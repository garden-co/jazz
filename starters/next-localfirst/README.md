# next-localfirst

A Next.js starter that ships as a pure local-first app out of the box — no
sign-in wall, no accounts, data synced to the Jazz server under a per-device
anonymous identity. A single env var flips on a [Better Auth](https://better-auth.com)
upgrade path that lets users opt in to a named account without losing their
existing data.

Think of it as a living tutorial for the local-first → hosted-auth upgrade
journey: both modes live in the same codebase, and the seam between them is
deliberately visible.

## Getting started

```bash
pnpm create jazz my-app --starter next-localfirst
cd my-app && pnpm dev
```

Open [http://localhost:3000](http://localhost:3000) and you'll land straight on
the app — a working todo list, no sign-in required.

No `.env` setup needed to get started. The `pnpm install` postinstall script
(`scripts/init-env.mjs`) generates `.env.local` with all required secrets, and
the `withJazz` plugin in `next.config.ts` starts a local Jazz dev server and
injects the Jazz env vars automatically on `next dev`.

## Pure local-first mode (default)

When `NEXT_PUBLIC_ENABLE_BETTERAUTH` is unset or `0`, the app runs in
local-first mode:

- The homepage is the app — there is no sign-in wall.
- Identity is derived from a per-device Ed25519 secret stored via
  `BrowserAuthSecretStore`. Each device gets its own anonymous Jazz user id.
- All data is synced to the Jazz server under that anonymous identity.
- There is no sign-out button and no concept of a named account.

This is a fully functional mode, not a degraded state. Most apps can ship with
this alone.

The `BETTER_AUTH_SECRET` is still generated by the postinstall script in this
mode, so the upgrade path is ready to activate whenever you want it.

## BetterAuth upgrade path

To activate named accounts, open `.env.local` and set:

```
NEXT_PUBLIC_ENABLE_BETTERAUTH=1
```

Restart `pnpm dev`. The app now grows:

- A "Sign up to access from any device" button in the header for anonymous users.
- A "Sign in" link for returning users.
- `/signup` and `/signin` routes that render email/password forms.
- A sign-out button and email display when authenticated.

### Identity continuity

The key design question is: what happens to the todos a user created
anonymously when they sign up for a named account?

The answer is nothing — they carry over automatically. Here is how:

1. At sign-up, the client calls `db.getLocalFirstIdentityProof(...)` to mint a
   short-lived proof token that cryptographically asserts the client's current
   Jazz user id.
2. The signup form includes this token in the request body.
3. `src/lib/auth.ts` intercepts the `/sign-up/email` request via a
   `hooks.before` middleware, calls `verifyLocalFirstIdentityProof` (from
   `jazz-napi`), and extracts the verified Jazz user id.
4. `databaseHooks.user.create.before` pins the new BetterAuth `user.id` to that
   Jazz user id instead of generating a fresh one.
5. All subsequent JWTs carry `sub: <jazz-user-id>`, so the server sees the same
   principal that created the anonymous todos. Same Jazz id, same data.

### What's always in the codebase

The auth code is unconditionally present — the flag only controls whether it
activates:

- `src/lib/auth.ts` — Better Auth server config with the identity proof hook.
- `src/lib/auth-client.ts` — Better Auth React client.
- `app/api/auth/[...all]/route.ts` — catch-all Better Auth handler.
- `app/signup/` — sign-up page and form.
- `app/signin/` — sign-in page and form.

## Architecture

```
app/
  layout.tsx                        ← plain root layout
  page.tsx                          ← app homepage (todo list + conditional header actions)
  providers.tsx                     ← JazzAuthProvider: resolves JWT or local-first secret
  api/auth/[...all]/route.ts        ← Better Auth handler (active when flag is on)
  signup/                           ← sign-up page (active when flag is on)
  signin/                           ← sign-in page (active when flag is on)
schema.ts                           ← Jazz app schema (todos table)
permissions.ts                      ← row-level access policy
src/
  lib/auth.ts                       ← Better Auth server config + identity proof hook
  lib/auth-client.ts                ← Better Auth React client
  components/todo-widget.tsx        ← Jazz-powered todo list
```

### Jazz integration

`JazzAuthProvider` in `app/providers.tsx` handles both modes transparently. On
mount it resolves the right credential:

- If BetterAuth is enabled and there is an active session, it fetches a JWT and
  passes it to `<JazzProvider>`.
- Otherwise it calls `BrowserAuthSecretStore.getOrCreateSecret()` and connects
  with the local-first secret.

`BetterAuthJazzSync` listens for `onAuthChanged` events and refreshes the JWT
when the token expires, so the Jazz connection stays live across long sessions.

### Extending the schema

Edit `schema.ts` to add tables. The Jazz dev server watches the file and
republishes the schema on change — no restart needed.

```ts
const schema = {
  todos: s.table({ title: s.string(), done: s.boolean() }),
  projects: s.table({ name: s.string() }),
};
```

Row ownership is enforced by `permissions.ts` via the `$createdBy` predicate —
no explicit `ownerId` column required.

## Environment variables

`.env.local` is generated by `scripts/init-env.mjs` on first `pnpm install`.
The script is idempotent — re-running `pnpm install` never overwrites an
existing `.env.local`.

| Variable                        | Default                    | Purpose                                               |
| ------------------------------- | -------------------------- | ----------------------------------------------------- |
| `BETTER_AUTH_SECRET`            | 32 random bytes, base64url | Required. `src/lib/auth.ts` throws loudly if missing. |
| `NEXT_PUBLIC_ENABLE_BETTERAUTH` | `0`                        | Set to `1` to activate the BetterAuth upgrade path.   |

`NEXT_PUBLIC_JAZZ_APP_ID` and `NEXT_PUBLIC_JAZZ_SERVER_URL` are injected at
runtime by the `withJazz` Next.js plugin during dev — no manual setup needed.

## Deploying to production

Two things to know before deploying:

The Jazz cloud server requires `--allow-local-first-auth` explicitly in
production. In development this flag is on by default; in production you must
pass it: `jazz-tools server <APP_ID> --allow-local-first-auth`. Without it,
anonymous local-first connections will receive auth errors.

`.env.local` is gitignored and not committed. Production deployments must
supply `BETTER_AUTH_SECRET` through your hosting provider's secret management.
The value must be consistent across restarts — rotating it invalidates all
existing Better Auth sessions.

## Known limitations

### Cross-device sign-in orphans anonymous data

If a user signs up on device A (which pins their BetterAuth account to Jazz
user id `X`), then opens the app fresh on device B (which creates a new
anonymous Jazz user id `Y`), creates todos on device B, and then signs in on
device B — the JWT carries `sub: X`. Any rows created under `Y` become
orphaned: they remain on the server but are inaccessible to the signed-in
principal, since policies match on `$createdBy`.

This is architectural: the current local-first auth model pins exactly one Jazz
id per BetterAuth account at sign-up. It is not trivially fixable inside a
starter. See
[`todo/issues/cross-device-signin-orphans-anonymous-data.md`](../../todo/issues/cross-device-signin-orphans-anonymous-data.md)
for the full analysis and possible directions.
