# expo-hybrid

An Expo starter for [Jazz](https://jazz.tools) that combines local-first
identity with optional [BetterAuth](https://www.better-auth.com/) sign-up.
Anonymous visitors get a per-device Jazz identity; signing up promotes the
same identity to a managed account, preserving any data created
anonymously.

## What this starter gives you

- A working Expo + Hono backend that runs end-to-end on first build.
- Anonymous local-first by default — the device boots straight into a
  working todo app with no sign-in.
- Optional sign-up via BetterAuth that promotes the local-first identity
  to a managed account using a `getLocalFirstIdentityProof` token, so
  data created anonymously is preserved.
- BetterAuth JWT minted on sign-in and handed to `<JazzProvider>` as the
  Jazz session token. Token refresh is wired up via `db.onAuthChanged`.
- A recovery-phrase backup widget so users can restore their local-first
  account if they sign out without signing up.

## Getting started

```bash
pnpm install
pnpm --filter expo-hybrid start
```

The `start` script runs the BetterAuth server on `:3001` and the Expo dev
server concurrently. To boot the native app on a simulator:

```bash
pnpm --filter expo-hybrid ios       # iOS simulator
pnpm --filter expo-hybrid android   # Android emulator
```

This app uses native code (`jazz-rn`), so it runs only as a development build
— **not** in Expo Go. The first run triggers `expo prebuild`.

The `prebuild` npm script generates a `.env` with a fresh
`BETTER_AUTH_SECRET` if one is not already present. The `withJazz` Metro
helper provisions `EXPO_PUBLIC_JAZZ_APP_ID` and
`EXPO_PUBLIC_JAZZ_SERVER_URL` on first start. Metro inlines them into the
bundle.

The Expo client picks the BetterAuth host based on `Platform.OS`:

- iOS simulator: `http://127.0.0.1:3001`
- Android emulator: `http://10.0.2.2:3001`
- Override with `EXPO_PUBLIC_AUTH_SERVER_URL` for a physical device.

## Architecture

```
src/
  todo-widget.tsx                ← Jazz-powered todo list (React Native)
  auth-backup.tsx                ← recovery-phrase backup/restore for anonymous users
  sign-in-form.tsx               ← email + password sign-in
  sign-up-form.tsx               ← sign-up with local-first identity proof
  auth-client.ts                 ← BetterAuth Expo client (secure-store session)
App.tsx                          ← app entry, mounts BetterAuth + Jazz providers
schema.ts                        ← Jazz app schema (todos table)
permissions.ts                   ← row-level access policy ($createdBy)
server/
  app.ts                         ← Hono router
  auth.ts                        ← BetterAuth instance with proof verification hook
  jwt-payload.ts                 ← JWT payload customisation hook
  index.ts                       ← Hono runtime entry
metro.config.mjs                 ← starts the Jazz dev server, injects env
```

## How it works

Every device gets its own Ed25519 secret, generated and stored by
`ExpoAuthSecretStore` on first launch. While anonymous, the app uses that
secret as the Jazz session secret directly.

When a user signs up, the client mints a short-lived
`getLocalFirstIdentityProof` token and posts it to
`/api/auth/sign-up/email`. A BetterAuth `before` hook on the server
verifies the proof using `verifyLocalFirstIdentityProof` from `jazz-napi`,
then sets the new BetterAuth user's `id` to the proven Jazz user id.
After sign-up, the same identity is now backed by a managed account, and
the Jazz client switches from `secret` to `jwtToken` mode.

`JwtRefresh` listens for Jazz auth-state changes; on `expired`, it requests
a new JWT and calls `db.updateAuthToken`.

## Extending the schema

Edit `schema.ts` to add tables. The Jazz dev server watches the file and
republishes the schema on change.

```ts
const schema = {
  todos: s.table({ title: s.string(), done: s.boolean() }),
  projects: s.table({ name: s.string() }),
};
```

Row ownership is enforced by `permissions.ts` via the `$createdBy` predicate,
so you don't need an explicit `ownerId` column.

## Environment variables

| Variable                      | When                 | Source                                                |
| ----------------------------- | -------------------- | ----------------------------------------------------- |
| `BETTER_AUTH_SECRET`          | always               | generated by `prebuild` script                        |
| `APP_ORIGIN`                  | always               | defaults to `http://localhost:3001`                   |
| `EXPO_PUBLIC_AUTH_SERVER_URL` | physical device only | manual                                                |
| `EXPO_PUBLIC_JAZZ_APP_ID`     | cloud only           | scaffolder (`create-jazz --hosting hosted`) or manual |
| `EXPO_PUBLIC_JAZZ_SERVER_URL` | cloud only           | scaffolder or manual                                  |
| `JAZZ_ADMIN_SECRET`           | cloud only           | scaffolder or manual                                  |
| `BACKEND_SECRET`              | cloud only           | scaffolder or manual                                  |

Leave the four cloud vars unset for self-hosted mode — the `withJazz` Metro
helper writes the first two automatically. For cloud mode, scaffold via
`create-jazz --hosting hosted` or paste values from
https://v2.dashboard.jazz.tools.

## Deploying to production

For cloud-hosted deployments, set the four cloud vars in your build
environment (EAS / your CI) and your app will sync against Jazz Cloud.

For self-hosted deployments you need to run your own Jazz server
alongside the Hono BetterAuth server. The auth server's base URL must be
set in `APP_ORIGIN`, and the Expo client must reach it via
`EXPO_PUBLIC_AUTH_SERVER_URL`.

The in-memory BetterAuth database is for development only — replace
`memoryAdapter` in `server/auth.ts` with a persistent adapter (Drizzle,
Prisma, etc.) before shipping. The self-hosted Jazz server requires
`--allow-local-first-auth` to accept anonymous local-first connections.

## Known limitations

- **In-memory BetterAuth database.** All accounts vanish when the server
  restarts. Swap in a persistent adapter for production.
- **Sign-up requires an active local-first session.** The proof token is
  minted from the device's local-first secret, so sign-up only works while
  the anonymous identity is loaded.
- **No native passkey backup.** The recovery-phrase flow is the only
  built-in backup option on Expo. Browser passkeys
  (`BrowserPasskeyBackup`) are not available in React Native.

## Where to go next

- `schema.ts` and `permissions.ts` — the two files you'll touch most when

- `server/auth.ts` — swap `memoryAdapter` for a persistent BetterAuth

- `src/todo-widget.tsx` — replace with your own UI.
