Add SSR query hydration: prefetch Jazz queries on the server, embed the resolved rows in the rendered HTML, and seed the client orchestrator so the first paint after hydration shows data with no loading flash or client-side refetch.

- `jazz-tools/backend`: new `createSnapshotBuilder({ appId, schema, principalId? })` with `prefetch(db, query, options?, session?)` and `dehydrate()` for capturing server-side query results into a serialisable envelope.
- `jazz-tools/react` / `jazz-tools/react-core`: `JazzProvider` accepts an optional `snapshot` prop (plus optional `schema` for envelope validation). The snapshot seeds a synchronous, read-only orchestrator and renders the prefetched rows on the first render — server-side included — instead of suspending on the async client, so the SSR HTML matches the client's first paint with no hydration re-render. Once the live client connects it is seeded with the same rows and swapped in; `useAll` reads from this seed-or-live orchestrator, while hooks that need the live client (`useDb`, `useSession`, `useAuthState`) suspend until it connects.
- `jazz-tools/svelte`: `JazzSvelteProvider` accepts the same optional `snapshot` (and `schema`) props and seeds synchronously in its script body, and `QuerySubscription` does a synchronous initial read — so server-rendered SvelteKit HTML and the client's first paint already contain the prefetched rows, then swap to the live client transparently once it connects.
- Snapshot validation: `appId` and `schemaFingerprint` mismatches discard the snapshot with a dev-mode warning and fall back to a live fetch. The principal is not a display gate — the client trusts a server-rendered snapshot and renders its rows on the first paint, user-scoped or not; keeping the snapshot scoped to the viewer (prefetch with the right `Db`, don't cache or serve it cross-principal) is the server's responsibility. As a loud backstop the client still throws if it holds a confirmed live principal that contradicts the snapshot's — a snapshot scoped to one user reaching a different logged-in client. `principalId` stays on the envelope as server-side provenance.
- `JazzProvider.schema` / `JazzSvelteProvider.schema` and `createSnapshotBuilder({ schema })` accept either the raw `WasmSchema` or your merged `app` directly — the `wasmSchema` field is unwrapped automatically.
- `SubscriptionsOrchestrator`: new `seedSnapshot(key, snapshot)` method and exported `computeQueryKey(appId, query, options)` helper so server-side code can derive the orchestrator cache key. `serializeQueryOptions` now uses a canonical (sorted-key) stringifier, so query option property order no longer affects cache identity.
- Fix update permission policies that compare `id` inside `EXISTS` checks, so session-scoped prefetches don't leak rows across principals.
