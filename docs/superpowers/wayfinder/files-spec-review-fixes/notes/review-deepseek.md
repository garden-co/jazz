Here is the adversarial review of the spec pair, one finding per line.

## Findings

1. **CRITICAL** | PRD:ImplDecisions(File ids), Slice-1:ImplDecisions(Id format) | File id concrete wire format is unspecified — encoding (base64? hex?), separator, and parsing algorithm are all absent | The server must extract the identity segment from the id to authorize grants, releases, and deletes; the SDK must parse it to construct `url()` and object keys. If two implementers invent incompatible formats, authorization bypass (segment-confusion attacks) is the failure mode, not just interop drift.

2. **CRITICAL** | PRD:ImplDecisions(Self-collision belt) | `If-None-Match: *` on `CompleteMultipartUpload` is not supported by standard S3 | The spec claims conditional multipart completion is a self-collision guard; standard S3's `CompleteMultipartUpload` does not accept conditional headers. This either breaks the S3-compatible contract or means the guard simply doesn't exist for multipart uploads, leaving an unacknowledged gap.

3. **CRITICAL** | PRD:ImplDecisions(Upload flow step 5), Slice-1:ImplDecisions(File-plane protocol) | Release message omits `file id` in PRD but requires it in Slice-1 | The PRD says release carries "the UploadId and part ETag list"; Slice-1 adds `(file id, UploadId, part ETags)`. The server needs the file id to derive the final object key and verify the identity segment. An implementer following only the PRD's flow ships a broken release message.

4. **CRITICAL** | PRD:ImplDecisions(Serving endpoint), PRD:ImplDecisions(Reads are URL-only) | `url()` is described as "pure local string construction from the id alone" but its base URL depends on deployment configuration (server redirect vs. CDN-direct) | The spec says "deployments may equally point a CDN straight at the bucket" — the URL host differs per deployment, so `url()` cannot be purely local without the client knowing the base. This contradicts the sync-and-local claim; the base URL must reach the client through configuration, and the spec never says how.

5. **MAJOR** | PRD:ImplDecisions(Canonical JSON), Slice-1:ImplDecisions(Validation) | Canonical JSON algorithm is undefined — key ordering rule, number formatting, string escaping | The descriptor's byte form affects equality and digests ("digests are byte-stable," PRD story 3). Two implementations applying different canonicalization produce different bytes for the same logical descriptor, breaking sync and hash-based operations. RFC 8785 or an explicit minimal rule must be named.

6. **MAJOR** | PRD:ImplDecisions(Serving), Slice-1:UserStories(18) | Inline-safe MIME type allowlist is not enumerated — only examples given (image, video, audio, PDF) and exclusions named (text/html, svg) | "Implementation-owned" is ambiguous: per-server? per-SDK? Different servers serve different types inline, creating divergent security postures. A concrete list is required for the content-type-abuse hardening to mean anything.

7. **MAJOR** | PRD:ImplDecisions(Authorization), Slice-1:ImplDecisions(File-plane protocol) | `{app}` segment in object keys is never assigned — how does the server know which app value to use for grant/release/delete? | The key is `{app}[/t{class}]/{identity}/{random}`. The sync connection is app-scoped, but the spec never states the derivation rule. Without it, a server might use a different app identifier than the SDK, producing key mismatches.

8. **MAJOR** | PRD:ImplDecisions(File ids), PRD:UserStories(38) | Identity-segment parsing safety depends on an unspecified delimiter | The spec claims takeover is "impossible by construction" because the server compares identity segments. If the delimiter can appear inside identity ids, parsing is ambiguous and the comparison can be subverted. The id format must use a delimiter forbidden in identity ids (or length-prefix or base64-encode components) — none of this is specified.

9. **MAJOR** | PRD:UserStories(4,6), PRD:FurtherNotes | "Copy descriptor = access bytes" is not listed among the stated accepted semantics | The design allows Alice to copy Bob's file id into her own cell; her authorized readers then have the URL and can fetch Bob's bytes, even if Bob's row policy excludes them. This is a direct consequence of public-read bytes + hand-rolled descriptors. It belongs in the "stated, accepted semantics" list alongside identity-linkability and the random-part-as-sole-barrier, but is absent.

10. **MAJOR** | PRD:ImplDecisions(Upload flow step 2), Slice-1:ImplDecisions(Outbox hold) | Outbox hold granularity is ambiguous — "causally dependent writes queue behind it" is never defined | What constitutes "causally dependent"? Same row? Same table? Referential integrity? The PRD says independent writes bypass but dependent ones queue. Without a definition, two SDKs will produce different queuing behavior, and apps relying on one will break on the other.

11. **MAJOR** | PRD:ImplDecisions(Upload flow step 5) | `CopyObject` from pending to final key must preserve object metadata (Content-Type, Cache-Control, Content-Disposition) but this is not stated | S3 CopyObject allows metadata replacement; if the copy doesn't explicitly carry over the pinned upload headers, the final object loses its security-hardening headers. The spec should mandate metadata preservation.

12. **MAJOR** | Slice-1:ImplDecisions(Delete execution) | "Backend surface bypasses" identity check for deletion but the backend surface is never defined | Is it an admin API key? A server-to-server internal call? An operator credential? Without a concrete definition, the delete authorization model has an undefined escape hatch that cannot be implemented or tested.

13. **MAJOR** | PRD:TestingDecisions, Slice-1:TestingDecisions | No test scenario covers the release-idempotency interaction with the outbox hold after crash | The PRD's explicit scenarios list "release idempotency" and "upload interrupted by client restart" separately, but the combination — restart during release (hold lost, release retry converges via HEAD) — is untested. This is the exact edge where in-memory state loss meets idempotent server convergence.

14. **MAJOR** | PRD:ImplDecisions(Lease window) | Lease window / `pending/` TTL is never pinned to a concrete value — "on the order of days" | The lifecycle rule needs a specific number of days. Different deployments picking different values (1d vs 7d) produce different abuse-window and resume-window tradeoffs with no guidance.

15. **MINOR** | PRD:ImplDecisions(Unreleased-upload cleanup) | "After lease expiry the SDK restarts the upload with a fresh id" is stated unconditionally but only works intra-session | After a restart the in-memory Blob is gone; there is nothing to restart from. The unannotated Implementation Decisions text doesn't qualify this, so an implementer could read it as a general restart-after-crash behavior, contradicting amendment 9.

16. **MINOR** | PRD:ImplDecisions(Single PUT threshold) | Multipart threshold is "tens-of-MB implementation constant" — no specific value | Different SDKs picking 10MB vs 50MB is functionally harmless but confusing in cross-platform testing where one side uses multipart and the other doesn't.

17. **MINOR** | PRD:ImplDecisions(TTL class set), Slice-1:ImplDecisions(TTL) | TTL class set configuration format is unspecified — env var? config file? CLI? | The server validates the class segment against "the deployment's class set" but no configuration surface is defined, forcing every implementer to invent one.

18. **MINOR** | PRD:ImplDecisions(Serving) | Pinned `Content-Type` comes from the client's `mime_type` in the descriptor, which is untrusted | A lying client uploads HTML with `mime_type: "image/png"`; the pinned Content-Type header serves it as `image/png`. `X-Content-Type-Options: nosniff` mitigates but doesn't eliminate the risk. The interaction between untrusted metadata and pinned headers is not discussed.

19. **MINOR** | PRD:ImplDecisions(FromBlob) | `fromBlob` takes a `Blob` but cross-platform Blob APIs differ — web `Blob`, Node `Buffer`, RN polyfill | The spec talks about "the Blob" as if it's a universal primitive; the three target platforms have different binary representations, and the `fromBlob` signature across bindings is not addressed.

20. **MINOR** | PRD:ImplDecisions(TTL clock), Slice-1:UserStories(22) | TTL clock starts at "the release copy" but lifecycle rules are day-granular — a file released at 23:59 on day N could expire effectively 1 day earlier than one released at 00:01 on day N+1 | The day-granularity caveat is mentioned but the up-to-24-hour variance in effective TTL is not called out in the stated accepted semantics, and app developers relying on exact lifetimes will be surprised.

21. **MINOR** | Both:UserStories | User story numbering is inconsistent between PRD (58 stories) and Slice-1 (25 stories) with no mapping table | Cross-referencing requires manual scanning; implementers using both docs will waste time aligning stories.

## Verdict

**Not yet ready for implementation.** The design is sound at the architectural level — the identity-bound key scheme, stateless authorization by string comparison, and the column-type-lowering-onto-text pattern are all well-conceived — but the spec pair has four critical gaps an agent must invent answers for (file id wire format, the S3-unsupported conditional multipart completion claim, the release message missing the file id, and the `url()` base-URL contradiction with the "purely local" claim). Another eight major findings cover underspecified security boundaries (the MIME allowlist, the `{app}` segment, the id delimiter), undefined operational surfaces (the backend delete surface, the TTL class-set config, the outbox hold's dependency definition), and a missing stated semantic (the copy-descriptor-equals-access-bytes pathway). None of these are design rethink — they are all specification gaps that can be resolved by writing down the concrete choices. Close the four criticals and the eight majors, and the spec pair is solid for implementation.
