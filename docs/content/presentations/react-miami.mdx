# Fresh Idea 1: Tunable consistency

<Notes>
The first idea is a simple compromise. In the old version
of Jazz, which I'll call classic Jazz from now on, we made
learned again and again through our adopters that, surprisingly,
in most apps, eventual consistency is fine 90% of the time.

Like it's OK if for a short moment, we see different versions
of shared data.

But the exceptions where it's not OK are important! like only one
user being able to buy the last ticket for an React Miami,
or wanting to take resources away here and deposit
them over there in an all-or-nothing way.

And again, we don't want to use a whole extra database just because
some corners of our app need global consistency like that, yuck!

So we found a clean way to just let you choose, for each part of your
app, for each table in your schema, whether it needs to be globally
consistent, or whether eventual consistency is OK. And like that you,
as the app developer can granularly trade off latency and offline support
with consistency. Because you as the app developer know best, and you
deserver something better than whole systems that force you into one
or the other for your whole app.

</Notes>
</Slide>

# What if every row had **git-like** history
- table 2D
- sync 3D (with tunable consistency and permissions)
- fluid schema evolution 4D
- per-row branching history 5D

<Notes>
But branching history, like git? In the simplest case
this makes the tricky problem of people editing data
at the same time actually tractible.

It's actually even better than git, because it lets you
choose a custom, precise merge strategy per column,
whether that's a simple last-writer wins, or a deep json
merge, or a smart rich-text merge.

This is another example where if you wanted any of this,
you would need to combine a bunch of different libraries
for collaboration, each of which would kind of force you
into one model. Here, you get everything in one system,
and you as the app developer who understands the best
collaboration behavior for your app can choose in detail.

</Notes>
