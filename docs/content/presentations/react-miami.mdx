# Metrics to judge a stack by

- Lines of code
- Layers of infra BS
- Light-milliseconds

<Notes>
And the three metrics I want you to keep in mind when
you judge a stack are

Lines of code, especially the ones that don't have anything to
do with what makes your app unique.

Layers of infra bullshit, meaning how many things you have to
deploy and integrate with each other.

Light-milliseconds that user edits have to travel before they
become visible.

</Notes>

<Notes>
The first thing these modern systems usually collapse is the frontend
sync glue.

Data fetching disappears as a concept.

The websocket client disappears as a concept.

The query cache and the optimistic update layer start getting absorbed
into one database-shaped client abstraction.

So the first win is not even mostly about infrastructure.

It is that the frontend starts feeling like it is working with
reactive data again instead of doing orchestration around remote state.

Lines of code go down immediately.

And on the light-milliseconds metric, we get the first huge win:
optimistic preview goes to zero.

This does not always save a dramatic amount of code by itself, because
one abstraction is often replacing another.

But it simplifies the shape of the system.

You stop having one database model on the server and a completely
different data access model on the client.

And psychologically this is an important step, because it normalizes
the idea that yes, a database client can live in the frontend too.

# Roughly where modern stacks land

Lines of code: ++++

Layers:

- backend/external
  - Database with auth policies
  - ~~ORM,~~ cache, ~~connection pooler~~
  - backend framework, DB client, ~~CRUD API boilerplate,~~ ~~API layer~~, ~~auth logic~~

  - ~~data fetching~~, ~~websocket client~~

Light milliseconds:

- full: 100ms -> backend -> 100ms client (no auth roundtrip!)
- optimistic preview: 0ms

<Notes>
And then permissions start moving inward too.

Instead of auth being something the backend wraps around every query,
the database itself starts understanding auth policies.

That means a whole chunk of backend auth glue disappears.

It also means one of the weirdest parts of the full request path
disappears: that extra auth roundtrip between the backend and the
database.

At this point the stack is already much, much cleaner than the one we
started with.

</Notes>

# Jazz pushes further: regional edges

Lines of code: ++++

Layers:

- backend/external
  - Database with auth policies
  - ~~cache~~
  - backend framework
  - workers
  - auth provider
- regional database edge
- frontend
  - DB client
  - ui state
  - client/rendering framework

Light milliseconds:

- full: 100ms -> backend -> 100ms client (no auth roundtrip!)
- via same regional cache: 10ms -> edge -> 10ms
- optimistic preview: 0ms

<Notes>
The next question is: if we have already made it this far, what else can
we do?

One answer is that we can get much more aggressive about latency.

So Jazz adds built-in regional edges for caching and for very low-latency
collaboration between nearby users.

That gives us another mode on the light-milliseconds metric:
not just full remote and not just optimistic preview, but genuinely
short regional paths.

And unlike a normal cache bolted onto the side, this thing is aware of
the data model and the auth model.

That matters, because otherwise these low-latency layers quickly turn
into yet another source of glue and inconsistency.

# Jazz pushes further: truly local-first

Lines of code: +++

Layers:

- backend/external
  - Database with auth policies
  - ~~cache~~
  - backend framework
  - workers
  - auth provider
- regional database edge
- frontend
  - **Local-first database**
  - ui state
  - client/rendering framework

Light milliseconds:

- full: 100ms -> backend -> 100ms client (no auth roundtrip!)
- via same regional cache: 10ms -> edge -> 10ms
- **durable local state**: 0ms

<Notes>
And this is where it gets really interesting.

We stop treating the client as just a fancy cache with optimism.

Instead, we introduce a truly durable local-first database on the client.

At that point the dream of `useState()` as your database starts becoming
real in a much stronger sense.

Not just reactive.

Not just optimistic.

Actually durable.

Actually queryable.

Actually part of the real system.

And that is the moment where the easy story ends and the hard story begins.

Because once the database is truly local-first, a whole new class of
problems shows up.

</Notes>
</Slide>

<Slide>
# Local-first got hands

- How do consistency and transactions work now?
- How do permissions work when data can exist locally?
- How do collaboration and history work?
- How does schema evolution work when old and new clients are all out there
  at once?

<Notes>
And these problems point at ways in which durable databases
that many users read from and write to
are simply truly different from ephemeral local state

</Notes>
