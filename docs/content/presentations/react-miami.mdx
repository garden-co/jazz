## Fresh Idea 1: Tunable consistency

<Notes>
The first idea is a simple compromise. In classic Jazz we learned again
and again through adopters that, surprisingly, eventual consistency is
fine for 90% of interactions in most apps.

It is usually OK if, for a short moment, different users see slightly
different versions of shared data.

But the exceptions matter: like only one user should be able to buy the
last ticket for React Miami.

And we don't want a whole second database just because some corners of
the app need global consistency.

So Jazz lets you choose, per table, whether **eventual** consistency is OK
or whether it needs **global** consistency, so you can trade latency and
offline support against consistency where it actually matters.

You as the app developer know best, and you shouldn't be limited to one

## Fresh Idea 2: Row-level security and per-query auth

<Notes>
The next idea is even more essential for making local-first work:
permissions.

In traditional backends, the boundary is clear: you enforce permissions
at the API in front of the database.

But here we have local state and direct sync, so the database itself has
to understand permissions.

For this we rely on a proven concept: row-level security. It lets you
define permissions as queries that look like your normal queries, in one
centralized, testable place.

That means the database can enforce permissions at sync time. Even
better, the local client also knows how permissions work and can preview
what the current user can and can't do before the roundtrip.

This is why I think row-level security is a good idea...

</Notes>

## Fresh Idea 3: Fluid schema evolution

<Notes>
Moving on, let me show you the most unique fresh idea of Jazz.

This is about the hardest part of local-first data:
changing the shape of your data as the app evolves.

You can't do a good old stop-the-world migration of a central database,
because there might still be old clients expecting the old shape.

Usually backwards compatibility is the API's job. See a pattern?
If clients sync directly to the database, the database needs to
understand backwards compatibility too.

So in Jazz we turned migrations into something slightly different.
Instead of a one-time event, migrations describe a mapping between two
schema versions.

And it's bidirectional, so we can translate both old data into the new
shape for new clients _and_ new data into the old shape for old clients.

Jazz does this just in time instead of making you stop the world.

</Notes>
</Slide>

<Slide className="justify-start">
## Fresh Idea 3: Fluid schema evolution

<Notes>
And then we realized something important:

Bidirectional migrations are not just a solution for local-first data.
They also make it much easier to deploy schema changes for _any_ app.

This same idea lets you work on ten features at once with your horde of
Claudes, or removes a central release bottleneck in a large company with
many teams launching different features behind flags.

Making all of that automatically compatible is suddenly much easier.

</Notes>
