<Slide className="justify-start">
## Metrics to judge a stack by

- Lines of code
- Layers of infra BS
- Light-milliseconds

<Notes>
And the three metrics I want you to keep in mind when you judge a stack
are:

Lines of code, especially the lines that do not make your app unique.

Layers of infra bullshit, meaning how many things you have to deploy and
integrate with each other.

And light-milliseconds that user edits have to travel before they
become visible.

</Notes>
</Slide>

## Slimming the stack 1

<img
  src="/presentations/react-miami/stacks/stack-1.svg"
  alt=""
  className="pointer-events-none absolute inset-0 z-0 h-full w-full object-cover"
/>

<Notes>
The first thing these modern systems usually collapse is the frontend
sync glue.

Data fetching and the websocket client disappear as concepts, and the
query cache plus optimistic update layer start getting absorbed into
one database-shaped client abstraction.

So the first great thing is that frontend starts feeling like it
is working with reactive data instead of orchestrating remote state.

Lines of code go down immediately, and on the light-milliseconds metric
we get the first huge win: optimistic preview goes to zero.

</Notes>
</Slide>

<Slide className="justify-start overflow-hidden [&_h2]:relative [&_h2]:z-10 [&_h2]:text-[3vw]">
## Slimming the stack 2

<img
  src="/presentations/react-miami/stacks/stack-2.svg"
  alt=""
  className="pointer-events-none absolute inset-0 z-0 h-full w-full object-cover"
/>

<Notes>
Next, the backend's database access story starts simplifying too.

The ORM or classic DB wrapper gets replaced by a more integrated
DB client.

This only saves a little amount of code, because
one abstraction is replacing another, but it simplifies the shape
of the system.

You stop having one database model on the server and a completely
different access model on the client, and it normalizes
the idea that yes, a database client can live in the frontend too.

</Notes>
</Slide>

<Slide className="justify-start overflow-hidden [&_h2]:relative [&_h2]:z-10 [&_h2]:text-[3vw]">
## Slimming the stack 3

<Notes>
Then some of the scale and performance crutches start falling off.

One obvious example is the connection pooler, because traditional

We're not removing much product code here,
but we are deleting operational surface area, which matters a lot.

  policy.letters.allowInsert.always();

<Notes>
And last, but most importantly, we have permissions.

Jazz actually makes you define them and fails closed otherwise. We also
tried to make the DSL readable enough that you can see at a glance
whether you set them up correctly.

For our app, everyone can create a canvas. Everyone can add letters and
move them around. But cursors can only be updated by the user that
created them. Makes sense, right?

And for more complex apps, Jazz gives you a permission testing framework
so you can set up tricky situations and edge cases without end-to-end
testing your whole app.
