Server-render a page that already contains your Jazz data, then let the client take over without refetching it on mount. The server runs the query through its backend `Db`, the result is dehydrated into a small JSON envelope, the envelope crosses the RSC boundary, and `<JazzProvider>` seeds its cache before the first child renders.

## What this is for

- Eliminating the empty-state flash on initial page load.
- Skipping the round-trip a client otherwise pays before sync delivers data.
- Keeping the rest of Jazz's local-first behaviour unchanged — once mounted, the client subscribes to the same query and sync corrects the snapshot to live state.

## What this is _not_ for

Read these before you reach for the API:

- **Catching the client up to the latest state at hydrate time.** The snapshot is a point-in-time read of resolved values; sync delivers fresher state after mount.
- **Server mutations.** This path is read-only — write through normal client APIs.
- **Streaming Suspense snapshots.** One envelope per render, captured before the response is sent.
- **Multi-principal pages.** One authenticated principal per snapshot.
- **Failed queries.** If a prefetch throws on the server, the snapshot just doesn't include that entry; the client refetches on mount.

## Server: prefetch a query

```ts title="lib/jazz-server.ts"
import { createJazzContext, createSnapshotBuilder } from "jazz-tools/backend";
import { app } from "../schema";
import permissions from "../permissions";

const context = createJazzContext({
  appId: process.env.NEXT_PUBLIC_JAZZ_APP_ID!,
  app,
  permissions,
  driver: { type: "memory" },
  serverUrl: process.env.NEXT_PUBLIC_JAZZ_SERVER_URL!,
  backendSecret: process.env.BACKEND_SECRET!,
});

// Build a fresh snapshot builder for every server render so prefetches
// don't bleed between requests.
export function createServerSnapshot() {
  return createSnapshotBuilder({
    appId: process.env.NEXT_PUBLIC_JAZZ_APP_ID!,
    schema: app,
  });
}

export { context };
```

In a Server Component, resolve a `Db` scoped to the right principal, run the queries the page renders, and dehydrate:

```tsx title="app/Todos.tsx"
import { app } from "@/schema";
import { context, createServerSnapshot } from "@/lib/jazz-server";
import TodosClient from "./TodosClient";

export default async function Todos() {
  // For per-user pages, swap this for a session-scoped Db so permissions
  // and per-principal visibility apply to the prefetch:
  //   const db = context.forSession(await readSession(...));
  const db = context.asBackend();

  const builder = createServerSnapshot();
  await builder.prefetch(db, app.todos);
  const snapshot = builder.dehydrate();

  return <TodosClient snapshot={snapshot} />;
}
```

## Client: seed JazzProvider

```tsx title="app/TodosClient.tsx"
"use client";

import { JazzProvider, useAll, type DehydratedSnapshot } from "jazz-tools/react";
import { app } from "@/schema";

export default function TodosClient({ snapshot }: { snapshot: DehydratedSnapshot }) {
  // `secret` is resolved client-side via BrowserAuthSecretStore — see
  // `examples/nextjs-csr-ssr/app/HydratedTodoClient.tsx` for the auth wiring.
  return (
    <JazzProvider
      config={{
        appId: process.env.NEXT_PUBLIC_JAZZ_APP_ID!,
        serverUrl: process.env.NEXT_PUBLIC_JAZZ_SERVER_URL!,
        secret,
      }}
      snapshot={snapshot}
    >
      <List />
    </JazzProvider>
  );
}

function List() {
  // Hits the seeded entry synchronously on first render — no loading state.
  const todos = useAll(app.todos) ?? [];
  return (
    <ul>
      {todos.map((t) => (
        <li key={t.id}>{t.title}</li>
      ))}
    </ul>
  );
}
```

The matching client-side `useAll(app.todos)` derives the same orchestrator key as the server's `prefetch(db, app.todos)`, finds the seeded entry, and returns the data synchronously. Sync still runs in the background and replaces the seeded data with live state once it arrives.

## SvelteKit

The same flow works in SvelteKit. Prefetch in a server `load`, return the envelope as page data, and pass it to `JazzSvelteProvider`.

```ts title="src/lib/jazz-server.ts"
import { createJazzContext, createSnapshotBuilder } from "jazz-tools/backend";
import { PUBLIC_JAZZ_APP_ID, PUBLIC_JAZZ_SERVER_URL } from "$env/static/public";
import { BACKEND_SECRET } from "$env/static/private";
import { app } from "$lib/schema";
import permissions from "$lib/permissions";

export const context = createJazzContext({
  appId: PUBLIC_JAZZ_APP_ID,
  app,
  permissions,
  driver: { type: "memory" },
  serverUrl: PUBLIC_JAZZ_SERVER_URL,
  backendSecret: BACKEND_SECRET,
});

// A fresh builder per request so prefetches don't bleed between renders.
export function createServerSnapshot() {
  return createSnapshotBuilder({ appId: PUBLIC_JAZZ_APP_ID, schema: app });
}
```

```ts title="src/routes/+page.server.ts"
import { app } from "$lib/schema";
import { context, createServerSnapshot } from "$lib/jazz-server";
import type { PageServerLoad } from "./$types";

export const load: PageServerLoad = async (event) => {
  // Public data, read as the backend principal. For per-user pages, scope the
  // read to the request instead — the rows still render in the SSR HTML:
  //   const db = await context.forRequest(event.request);
  const db = context.asBackend();

  const builder = createServerSnapshot();
  await builder.prefetch(db, app.todos);
  return { snapshot: builder.dehydrate() };
};
```

On the client, create the live client in the browser only — on the server the provider renders from the snapshot alone — and pass `snapshot` (plus `schema`, for envelope validation) to the provider:

```svelte title="src/routes/+page.svelte"
<script lang="ts">
  import { browser } from "$app/environment";
  import { createJazzClient, JazzSvelteProvider } from "jazz-tools/svelte";
  import { PUBLIC_JAZZ_APP_ID, PUBLIC_JAZZ_SERVER_URL } from "$env/static/public";
  import { app } from "$lib/schema";
  import TodoList from "$lib/TodoList.svelte";

  let { data } = $props();

  // No live client during SSR; the snapshot drives the server render. The
  // browser creates the real client, which then takes over.
  const client = browser
    ? createJazzClient({ appId: PUBLIC_JAZZ_APP_ID, serverUrl: PUBLIC_JAZZ_SERVER_URL })
    : new Promise<never>(() => {});
</script>

<JazzSvelteProvider {client} snapshot={data.snapshot} schema={app}>
  {#snippet children()}
    <TodoList />
  {/snippet}
</JazzSvelteProvider>
```

```svelte title="src/lib/TodoList.svelte"
<script lang="ts">
  import { QuerySubscription } from "jazz-tools/svelte";
  import { app } from "$lib/schema";

  // Hits the seeded entry synchronously — the rows are already in the SSR HTML.
  const todos = new QuerySubscription(app.todos);
</script>

<ul>
  {#each todos.current ?? [] as todo (todo.id)}
    <li>{todo.title}</li>
  {/each}
</ul>
```

`QuerySubscription(app.todos)` derives the same orchestrator key as the server's `prefetch(db, app.todos)`, reads the seeded entry synchronously on the server and on the first client render, then swaps to live sync once the browser client connects.

## Session delegation: who runs the query

The hardest part of server-rendering with a sync engine is "as whom does the server read?". Jazz exposes three options on `JazzContext`:

- **`context.asBackend()`** — reads as the backend principal, which can see everything, so only prefetch genuinely public data this way (marketing pages, leaderboards). Its snapshot carries a `null` `principalId` (no viewer scoping). Writes are attributed to `jazz:system`.
- **`await context.forRequest(req)`** — reads as the authenticated user behind the request, taking bearer tokens straight from HTTP headers. [Permission policies](/docs/auth/permissions) apply as if the user themselves were reading. Use for per-user app shells.
- **`context.forSession(session)`** — same as `forRequest`, but for when you already hold a resolved [`Session`](/docs/auth/sessions) (e.g. after running your own JWT middleware).

The snapshot the client hydrates from is only as well-scoped as the `Db` the server prefetched against. Pick the one whose visibility matches what the client will see post-hydration, otherwise the hydrated render and the post-sync render disagree, which the user reads as a flicker.

<Callout type="warn">
  Getting this wrong is a data-exposure bug, not just a flicker. If you prefetch with a `Db` that
  can see more than the viewing user — `asBackend()` for a per-user page, or a session scoped to the
  wrong principal — those rows are dehydrated into the envelope and shipped to the browser,
  delivering data the user may have no right to read. Match the prefetch scope to the viewer, and
  see [access control](/docs/recipes/access-control/user-owned-data) for per-user visibility.
</Callout>

## The envelope

`dehydrate()` returns a plain JSON value. The shape mirrors the `DehydratedSnapshot` exported from `jazz-tools/backend` (re-exported by `jazz-tools/react`) — import that type directly rather than redeclaring it locally:

```ts
import type { DehydratedSnapshot } from "jazz-tools/backend";

// DehydratedSnapshot = {
//   appId: string;             // must match the client config's appId
//   principalId: string | null; // who the prefetch ran as — server-side
//                               // provenance; the client doesn't gate display on it
//   schemaFingerprint: string;  // must match the client's schema (when `schema` is set)
//   entries: { key: string; result: unknown }[];
// };
```

`<JazzProvider>` validates the envelope. If `appId` or `schemaFingerprint` mismatch it **silently discards the snapshot (with a dev-mode console warning) and falls back to a normal fetch** — nothing breaks, the optimisation just doesn't kick in. The principal, though, is **not** a display gate: the client trusts the snapshot and renders its rows on the first paint, user-scoped or not. Keeping the snapshot matched to the viewer — prefetching with the right `Db`, and not caching or serving one principal's render to another — is the server's responsibility (the rendered HTML carries that data regardless of what the client does with the envelope). The one principal check the client _does_ make is a loud backstop: if it has a **confirmed** live principal that contradicts the snapshot's — a snapshot scoped to one user reaching a different logged-in client — it **throws** rather than silently swapping in another user's rows.

Pass your `schema` prop too if you want the fingerprint check to run:

```tsx
import { app } from "@/schema";

<JazzProvider config={...} snapshot={snapshot} schema={app}>
```

Without `schema`, the fingerprint check is skipped — you take responsibility for keeping server and client schemas in sync.

## Constraint: matching `QueryBuilder` expressions

The cache key is derived from the query, its serialised options, and your `appId`. (`query._build()` is part of the internals — not a public API you call, and not something you set directly.) Identical query expressions and options on both sides produce identical keys. **The expressions must match.** A server-side `app.todos.where({done: true})` and a client-side `app.todos` are different keys; the seeded entry won't be found.

If you find yourself wanting "any matching subset" semantics (return the seeded data even when keys don't exactly align), that's not what this gives you. Use the same query expression both places, or accept the cache miss.

## ISR and long-lived caching

If the page is statically rendered or sits behind a long CDN TTL, the snapshot embedded in the HTML is older than what sync will deliver. The client will briefly render the stale snapshot, then sync corrects it. Acceptable for most pages; not acceptable for pages that display adversarially-stale state (live scores, stock prices). For those, render dynamically or skip the snapshot entirely.

<Callout type="warn">
  Caching cuts the other way for **scoped** data. A snapshot prefetched as a specific user bakes
  that user's rows into the page's HTML, so if that page is statically rendered, held in the
  full-route cache, or sits behind a shared CDN, another user can be served the first user's data.
  Only publicly cache pages whose snapshot is public (`null` `principalId`). For per-user pages,
  render dynamically and keep the response private (e.g. `Cache-Control: private, no-store`).
</Callout>

<Callout type="info">
  The snapshot is JSON crossing the RSC boundary. Session secrets, backend secrets, and other
  server-only state never appear in it — only resolved query results and the envelope metadata
  above.
</Callout>
