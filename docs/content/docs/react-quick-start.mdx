## Install

```bash
npm install jazz-tools
```

## Define your schema

Create `schema/current.ts` at the root of your project. This is the source of truth for your data model.

```ts title="schema/current.ts"
import { table, col } from "jazz-tools";

table("projects", {
  name: col.string(),
});

table("todos", {
  title: col.string(),
  done: col.boolean(),
  description: col.string().optional(),
  project: col.ref("projects").optional(),
});
```

### Available column types

There are a number of column types available which cover most common use cases:

| DSL | SQL type | TypeScript type |
|-----|----------|-----------------|
| `col.string()` | TEXT | `string` |
| `col.boolean()` | BOOLEAN | `boolean` |
| `col.int()` | INTEGER | `number` |
| `col.float()` | REAL | `number` |
| `col.ref("table")` | UUID (FK) | `string` (row ID) |

Any column can be made nullable with `.optional()`.

### Run codegen

The codegen will take your schema definition and turn it into the appropriate SQL, TypeScript interfaces, and all the tools you need to build your app.

```bash
npx jazz-tools build
```

#### Output:
- `schema/current.sql` — the equivalent SQL
- `schema/app.ts` — typed interfaces, query builders, and the `app` object you import everywhere

<Callout title="Generated code">
  Never edit `schema/app.ts` by hand. Edit `schema/current.ts` and re-run `npx jazz-tools build`.
</Callout>

The generated `app` export gives you a typed entry point for every table:

```ts
import { app } from "../schema/app.js";

app.todos          // query builder for the todos table
app.projects       // query builder for the projects table
```

## Wrap your app in JazzProvider

`JazzProvider` initialises the Jazz runtime (loads the WASM core, opens local storage in OPFS, connects to the sync server) and makes the database available to all descendant components via React context.

```tsx title="src/App.tsx"
import { JazzProvider } from "jazz-tools/react";

function App() {
  return (
    <JazzProvider
      config={{
        appId: "my-app",
        serverUrl: "http://127.0.0.1:4200",  // optional — omit for offline-only
        jwtToken: "...",                       // optional — see Auth below
      }}
      fallback={<p>Loading...</p>}
    >
      <TodoList />
    </JazzProvider>
  );
}
```

### DbConfig reference

| Field | Required | Description |
|-------|----------|-------------|
| `appId` | yes | Application identifier. Isolates storage between apps. |
| `serverUrl` | no | Jazz sync server URL. Omit for fully local/offline mode. |
| `jwtToken` | no | JWT Bearer token for authenticating with the server. |
| `adminSecret` | no | Admin secret for schema catalogue sync. |
| `dbName` | no | OPFS database name (browser). Defaults to `appId`. |
| `env` | no | Environment label (e.g. `"dev"`, `"prod"`). |
| `userBranch` | no | Branch name. Defaults to `"main"`. |

`config` is read once at mount. If you need to change it (e.g. after login), remount the provider with a new `key`.

## Read data with useAll

`useAll` subscribes to a query and returns a live array that re-renders your component whenever the results change. It is the primary way to read data in React.

```tsx title="src/TodoList.tsx"
import { useAll } from "jazz-tools/react";
import { app } from "../schema/app.js";

function TodoList() {
  const todos = useAll(app.todos);
  //    ^? Todo[]  — always defined, starts as []

  return (
    <ul>
      {todos.map((todo) => (
        <li key={todo.id}>{todo.title}</li>
      ))}
    </ul>
  );
}
```

### Filtering, sorting, pagination

Query builders are fluent and immutable. Chain `.where()`, `.orderBy()`, `.limit()`, and `.offset()` to shape the query. The full query runs inside the local WASM runtime, so filtering is instant.

```tsx
// Only incomplete todos, alphabetical, max 50
const todos = useAll(
  app.todos
    .where({ done: false })
    .orderBy("title", "asc")
    .limit(50)
);
```

<Callout title="Fluent and immutable?">
  When you are building queries, you can create complex queries by chaining methods (fluency). 

  Each chained call returns a new query object rather than modifying the original. So if you do `const base = query.where({ active: true })` and then `const sorted = base.orderBy("name")`, `base` is unchanged (immutability), and you have two separate queries.
</Callout>

### Where operators

Simple equality uses a plain value. For other comparisons, pass an operator object:

```tsx
// Equality (shorthand)
app.todos.where({ done: false })

// Explicit operators
app.todos.where({ title: { contains: "milk" } })
app.todos.where({ project: { ne: someProjectId } })
```

Available operators by type:

| Type | Operators |
|------|-----------|
| string | `eq`, `ne`, `contains` |
| boolean | plain value only |
| number | `eq`, `ne`, `gt`, `gte`, `lt`, `lte` |
| timestamp | `eq`, `gt`, `gte`, `lt`, `lte` |
| uuid | `eq`, `ne`, `in` |
| ref (FK) | `eq`, `ne`, `isNull` (when nullable) |

By default `useAll` returns results immediately from the local replica (empty array until the first sync completes). You can optionally wait for a specific [persistence tier](#persistence-tiers) before delivering data.

## Write data with useDb

`useDb` returns the `Db` handle for mutations. All mutations are *synchronous*, although sync is not: they write to the in-memory WASM runtime instantly and sync to the server in the background.

```tsx title="src/TodoList.tsx"
import { useDb, useAll } from "jazz-tools/react";
import { app } from "../schema/app.js";

function TodoList() {
  const db = useDb();
  const todos = useAll(app.todos);

  // Insert — returns the new row's ID
  function addTodo(title: string) {
    const id = db.insert(app.todos, { title, done: false });
  }

  // Update — partial, only the fields you pass
  function toggleTodo(todo: { id: string; done: boolean }) {
    db.update(app.todos, todo.id, { done: !todo.done });
  }

  // Delete
  function removeTodo(id: string) {
    db.deleteFrom(app.todos, id);
  }

  // ... render
}
```

Because writes hit the local runtime first, the `useAll` subscription fires immediately with the updated data. There is no round trip to the server before the UI updates when using the default tier (`worker`).

When you need to confirm a write has reached a specific durability level before proceeding, see [persistence tiers](#persistence-tiers).

## Persistence tiers

Jazz data propagates through a series of tiers, from the fastest (local) to the most durable (central server). Both reads and writes can specify which tier to wait for.

| Tier | Where | When to use |
|------|-------|-------------|
| `"worker"` | Browser OPFS (local) | Default. Optimistic updates, data survives page refresh. |
| `"edge"` | Nearest sync server | Confirm data has left the device. Loading spinners, close-app guards. |
| `"core"` | Central server | Globally consistent state. Authoritative reads and writes. |

<Callout title="Which tier to use?">
  `worker` (or no tier) is the right default for most UI interactions. Use `edge` when you need to confirm data has synced to the nearest server, e.g. for loading indicators or guarding against tab closure. Use `core` when you need globally consistent state. You can set tiers on a per-query basis, so choose the right level for each operation.
</Callout>

### Reading with a tier

Pass a tier as the second argument to `useAll`. The return type becomes `T[] | undefined`, where `undefined` means the tier hasn't responded yet:

```tsx
const todos = useAll(app.todos, "edge");
//    ^? Todo[] | undefined  — undefined until the edge server confirms
```

### Writing with a tier

Use the `*Persisted` variants when you need a durability guarantee before continuing. The promise resolves once the specified tier acknowledges the write:

```tsx
const id = await db.insertPersisted(app.todos, { title: "Important", done: false }, "edge");
await db.updatePersisted(app.todos, id, { done: true }, "edge");
await db.deleteFromPersisted(app.todos, id, "core");
```

## Querying in depth

The examples above use `useAll` for reactive queries in components. Jazz also exposes imperative query methods on the `db` object for use outside the render cycle or outside of React.

### One-shot queries

```tsx
const db = useDb();

// All matching rows (async — reads from local storage)
const todos = await db.all(app.todos.where({ done: false }));

// First match or null
const todo = await db.one(app.todos.where({ id: someId }));
```

Outside of components (or when you need the delta), use `db.subscribeAll` directly. The callback receives a delta with the full result set plus granular add/update/remove lists:

```ts
const unsubscribe = db.subscribeAll(
  app.todos.where({ done: false }),
  (delta) => {
    delta.all;     // Todo[]  — full current result set
    delta.added;   // { item: Todo, index: number }[]
    delta.updated; // { oldItem: Todo, newItem: Todo, oldIndex: number, newIndex: number }[]
    delta.removed; // { item: Todo, index: number }[]
  },
);

// Later:
unsubscribe();
```

`useAll` is built on `subscribeAll` internally, so they have identical semantics.

### Relations with include

If your schema has `col.ref()` columns, codegen produces typed include interfaces. Use `.include()` to load related rows in a single query:

```ts
// Schema reminder:
// table("todos", {
//   project: col.ref("projects").optional(),
//   parent: col.ref("todos").optional(),   // self-referential
// });

// Load each todo's project and parent in one shot
const todos = useAll(
  app.todos.include({ project: true, parent: true })
);

// todos[0].project → Project | undefined
// todos[0].parent  → Todo | undefined
```

**Reverse relations** are generated automatically. A `project` FK on `todos` creates a `todosViaProject` reverse relation on `projects`:

```ts
const projects = useAll(
  app.projects.include({ todosViaProject: true })
);

// projects[0].todosViaProject → Todo[]
```

You can nest includes and even pass a query builder to filter related rows:

```ts
// Nested: load project, and for each project, load its todos
const todos = useAll(
  app.todos.include({
    project: {
      todosViaProject: true,  // nested include
    },
  })
);

// Filtered: only include incomplete child todos
const todos = useAll(
  app.todos.include({
    todosViaParent: app.todos.where({ done: false }),
  })
);
```

### Chaining multiple where calls

`.where()` calls are additive. Multiple calls produce AND semantics:

```ts
const results = useAll(
  app.todos
    .where({ done: false })
    .where({ title: { contains: "docs" } })
);
// Equivalent to: done = false AND title LIKE '%docs%'
```

### Combining everything

```ts
const results = useAll(
  app.todos
    .where({ done: false, title: { contains: "docs" } })
    .include({ project: true, parent: true })
    .orderBy("title", "asc")
    .limit(20)
    .offset(0)
);
```

## Authentication

Jazz uses JWT for authentication. Your app obtains tokens through whatever auth system you already have; Jazz just needs the token string passed into the provider config.

### How it works

1. Your app authenticates the user (your own login flow, OAuth, etc.)
2. Your backend issues a JWT with a `sub` claim identifying the user
3. You pass the token to `JazzProvider` via `config.jwtToken`
4. Jazz sends it as `Authorization: Bearer <token>` on all sync requests

The JWT payload Jazz expects:

```json
{
  "sub": "user-123",
  "claims": { "role": "admin", "team": "eng" },
  "exp": 1735689600
}
```

- `sub` (required) — becomes the user identity for this session
- `claims` (optional) — arbitrary JSON, forwarded to the server for policy evaluation
- `exp` (optional) — standard expiry, validated server-side

### Server-side configuration

The Jazz server validates tokens using either a shared HMAC secret (development) or a JWKS endpoint (production):

```bash
# Development: shared secret (HS256)
jazz-tools server $APP_ID --jwt-secret "my-secret" --admin-secret "my-secret"

# Production: JWKS endpoint (RS256)
jazz-tools server $APP_ID --jwks-url "https://auth.example.com/.well-known/jwks.json" --admin-secret "..."
```

Or via environment variables: `JAZZ_JWT_SECRET`, `JAZZ_JWKS_URL`, `JAZZ_ADMIN_SECRET`.

### Development shortcut

For local development, you can sign tokens client-side with the same shared secret the server uses:

```ts
async function signDevJwt(userId: string, secret: string): Promise<string> {
  const header = base64url(JSON.stringify({ alg: "HS256", typ: "JWT" }));
  const payload = base64url(JSON.stringify({
    sub: userId,
    claims: {},
    exp: Math.floor(Date.now() / 1000) + 3600,
  }));
  const enc = new TextEncoder();
  const key = await crypto.subtle.importKey(
    "raw", enc.encode(secret),
    { name: "HMAC", hash: "SHA-256" },
    false, ["sign"],
  );
  const sig = await crypto.subtle.sign("HMAC", key, enc.encode(`${header}.${payload}`));
  return `${header}.${payload}.${base64url(new Uint8Array(sig))}`;
}
```

Then pass it to the provider:

```tsx
const token = await signDevJwt("alice", "my-secret");

<JazzProvider config={{ appId: "my-app", serverUrl, jwtToken: token, adminSecret: "my-secret" }} />
```

<Callout title="Why adminSecret?">
  Clients currently need to push schema definitions to the server's catalogue, which requires the admin secret. This requirement will go away in a future release.
</Callout>

### No auth / offline only

Omit both `serverUrl` and `jwtToken` for a fully local, offline-only database. Data lives in the browser's OPFS and never leaves the device:

```tsx
<JazzProvider config={{ appId: "my-app" }} />
```

### Changing tokens

`config` is read once at mount. To handle token refresh or login/logout, remount the provider with a new React `key`:

```tsx
<JazzProvider key={jwtToken} config={{ ...config, jwtToken }}>
  {children}
</JazzProvider>
```
