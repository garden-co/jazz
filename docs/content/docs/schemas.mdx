Every Jazz app defines its schema in a root `schema.ts`, including Rust backends.

Each top-level property of the `schema` object is a table. In the example above, `projects` and `todos` become tables, and each nested property becomes a column on that table.

The `type AppSchema = s.Schema<typeof schema>; export const app: s.App<AppSchema> = s.defineApp(schema);` pattern is there to keep TypeScript hovers short and readable. Without that alias boundary, TypeScript tends to expand the entire schema literal when you hover `app`, `app.todos`, or derived row/query types.

## Lifecycle

- `schema.ts` defines structure. Structural edits create new schema hashes.
- `permissions.ts` defines the current auth rules. Permission-only edits do not create schema hashes.
- `migrations/` connect structural schema hashes so old and new shapes stay interoperable.

## Datatypes and References

- Use `s.string()`, `s.boolean()`, `s.int()`, `s.timestamp()`, and `s.bytes()` for scalar fields.
- Use `s.ref("tableName")` for foreign keys. Jazz infers typed `.include(...)` relations directly from `schema.ts`.
- For browser files and large uploaded blobs, define the conventional `files` / `file_parts` tables and store a normal reference to `files` from your app rows. See [Files & Blobs](/docs/files-and-blobs).

## Schema Defaults

You can declare default values directly in the schema definition with `.default(...)`.

```ts
import { schema as s } from "jazz-tools";

const schema = {
  todos: s.table({
    title: s.string(),
    done: s.boolean().default(false),
    tags: s.array(s.string()).default([]),
    ownerId: s.ref("users").optional().default(null),
  }),
};
```

Schema defaults are used in two places:

- For inserts, when a caller omits a field.
- In auto-generated lens defaults, when a column is added or removed across schema versions.

### Insert behavior

When you use insert new values:

- Omitted fields use the schema default if present.
- Fields explicitly set to `undefined` are treated the same as omitted.
- `null` is only allowed for nullable columns, and it is not replaced by the default value.

That means schema defaults affect future inserts, not reads of already-stored rows.

### Schema evolution behavior

Schema defaults also feed Jazz's auto-generated lenses:

- When a column is added across schema versions, Jazz uses the column's default for new rows that predate the new column.
- When a column is removed across schema versions, Jazz uses the column's default when auto-generating the reverse/default lens behavior for older schemas.

## Workflow and Schema Evolution

### Edit `schema.ts` during development

Most of the time, schema evolution starts with a normal edit to `schema.ts`. New writes use the new schema immediately.

If you only changed `permissions.ts`, stop there structurally. That updates the active permission bundle and does not require a migration. Optionally double-check locally with `npx jazz-tools validate`, inspect with `npx jazz-tools permissions status`, then publish with `npx jazz-tools permissions push`.

When older rows are no longer reachable from the current schema graph, Jazz logs a warning with the exact hashes you need next. These warnings can appear in browser logs, backend logs, or be forwarded from upstream servers. A typical warning looks like this, wrapped for readability:

```text
[client] Detected 3 rows of todos with differing schema versions.
[client] To ensure data visibility and forward/backward compatibility please create
         a new migration with:
         npx jazz-tools migrations create \
           a01f5c72ec47 \
           311995e9a178
```

That warning is the handoff from day-to-day schema editing to reviewed compatibility work.

### Generate a migration edge with `migrations create`

`migrations create` is only for structural schema changes. It connects to your Jazz server, loads both schema hashes, and writes a prefilled file into `migrations/{dateCreated}-unnamed-{fromHash}-{toHash}.ts`.

```bash
npx jazz-tools migrations create <fromHash> <toHash>
```

The generated file already contains `fromHash`, `toHash`, and minimal typed `from` / `to` witnesses. In most cases, the only manual work is to fill in `migrate` and replace `unnamed` in the filename with a real migration name.

### Publish a reviewed edge with `migrations push`

Publishing is the explicit gate. New schema hashes are still learned automatically when clients connect, but Jazz never auto-pushes migrations.

```bash
npx jazz-tools migrations push <fromHash> <toHash>
```

Run this after reviewing the migration file. Once pushed, that edge becomes available to make data visible across those schema hashes.

Backwards defaults define what older-schema clients see for fields dropped in newer schemas. When new clients write rows without that field, the backward lens supplies the configured default to old clients. If the lens is auto-generated, Jazz first looks for schema defaults on the column definition before falling back to heuristic defaults.

## Permissions

To define row-level policies, add a sibling `permissions.ts` next to `schema.ts`.

Those policies are versioned separately from structural schema hashes, so reviewing or publishing a permission change does not require a migration edge.
