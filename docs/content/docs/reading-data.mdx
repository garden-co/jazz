## Filters

### API Reference

- **Behavior:** `where(...)` adds AND-combined predicates to a query.
- **Parameters:** pass a partial row-shaped object.
- **TypeScript operators:** scalar equality plus typed operator objects by column kind.
- **Rust operators:** use explicit builder methods such as `filter_eq(...)`.

## Sorting

### API Reference

- **Behavior:** `orderBy(...)` applies deterministic ordering to query results.
- **Parameters:** `orderBy(column, direction?)` where `direction` is `"asc"` or `"desc"` in TypeScript.
- **Recommendation:** always sort before paginating.

## Pagination

### API Reference

- **Behavior:** `limit(...)` caps rows; `offset(...)` skips rows.
- **Parameters:** choose a stable page size and compute offset from page index.
- **Recommendation:** combine with a deterministic `orderBy(...)` so page boundaries are stable.

## Read Settled Tiers

### API Reference

- `db.all(query, settledTier?)`
- `db.one(query, settledTier?)`
- `db.subscribeAll(query, callback, settledTier?)`
- `settledTier`: `"worker" | "edge" | "core"`

Settled tiers control when initial query delivery is considered confirmed:

- `worker`: wait for worker/local storage acknowledgement.
- `edge`: wait for edge server acknowledgement.
- `core`: wait for core server acknowledgement.

Without a `settledTier`, reads are local-first and return immediately from local runtime state.

Tradeoff summary:

- Lower settled tier: lower latency, narrower confirmation scope.
- Higher settled tier: higher latency, broader confirmation scope.

- Use write ack tiers to gate mutation durability checkpoints.
- Use settled tiers to gate query/subscription delivery checkpoints.
