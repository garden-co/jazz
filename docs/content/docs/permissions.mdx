## Lifecycle

- `schema.ts` defines the structural data model. It produces schema hashes and is the input to migrations.
- `permissions.ts` defines the current auth rules for that app. Permission-only changes do not create schema hashes or migration edges.
- `migrations/` only describe structural compatibility between schema hashes.

Jazz applies the latest published permissions even when data is being read through older schema versions. Older rows are transformed into the current permission schema for auth checks, then projected back into the caller's requested shape.

## When Permissions Change

Editing only `permissions.ts` updates the active auth rules. It does not require `migrations create` or `migrations push`.

Reach for migrations only when the structure in `schema.ts` changes and older/newer schema hashes need a reviewed compatibility edge.

## Policy Enforcement and Request Context

Jazz permissions use relationship-based policy evaluation with requester session context. Backend endpoints should scope a client/session per request before reads and writes.
