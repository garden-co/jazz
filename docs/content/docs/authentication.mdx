Jazz 2 supports three auth methods:

1. `anonymous`: one synthetic user token per device
2. `demo`: multiple synthetic users per device for local multi-user simulation
3. `external`: bearer JWT validated via JWKS

## 1. Start with Anonymous Auth (Zero Config)

Browser clients default to anonymous local auth when no JWT/backend auth is configured.
On first load, Jazz auto-generates a device token and persists it in `localStorage`.

Use whatever column matches your schema, such as `owner_id`, `author_id`, `assignee_id`, or
`user_id`. The same pattern works whether a row is strictly owned by one user or simply scoped
to a user.

These client-side filters are useful for UX, but they do not make data private on their own.
Use [Permissions](/docs/permissions) to define row-level policies that actually enforce which
sessions can read or mutate a row.

## Session Resolution Order

For `jazz-tools server`:

1. Backend impersonation (`X-Jazz-Backend-Secret` + `X-Jazz-Session`)
2. External JWT (`Authorization: Bearer <jwt>`)
3. Local auth (`X-Jazz-Local-Mode` + `X-Jazz-Local-Token`)
4. No session

## Server Setup

### `jazz-tools server` (single app)

- Supports external JWT only when `--jwks-url` is configured.
- Local auth mode defaults depend on `NODE_ENV`:
  - `NODE_ENV=production`: anonymous/demo are disabled unless explicitly enabled with `--allow-anonymous` / `--allow-demo`
  - non-production (`development`, unset, etc.): anonymous and demo are enabled by default
- Supports external identity mapping persistence via `external_identities`.
