Jazz synchronizes data automatically across all connected clients and servers. This page explains how sync works.

## Sync Architecture

```
┌─────────────────┐           ┌─────────────────┐
│    Client A     │           │    Client B     │
│  ┌───────────┐  │           │  ┌───────────┐  │
│  │ IndexedDB │  │           │  │ IndexedDB │  │
│  └───────────┘  │           │  └───────────┘  │
└────────┬────────┘           └────────┬────────┘
         │                             │
         └──────────┬──────────────────┘
                    ↓
              ┌───────────┐
              │   Sync    │
              │  Server   │
              └─────┬─────┘
                    ↓
              ┌───────────┐
              │  Storage  │
              │ (RocksDB) │
              └───────────┘
```

## How It Works

1. **Local mutations** are applied immediately to local state
2. **Commits** are created for each mutation
3. **Sync server** receives commits and broadcasts to other clients
4. **Merge** occurs automatically when concurrent commits are received

## Conflict Resolution

When two clients edit the same row concurrently, Jazz uses per-column last-writer-wins:

- Each column tracks its last modification timestamp
- When merging concurrent edits, the most recent value for each column wins
- Changes to different columns merge cleanly (no conflict)
- Ties are broken deterministically by commit ID
