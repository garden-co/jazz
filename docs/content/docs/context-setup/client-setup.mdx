In browser clients, omitting `localAuthMode` and `localAuthToken` defaults to anonymous synthetic auth with a persisted local token. Omit `serverUrl` for a fully local/offline client.

## Where zero-config works

- The browser quickstarts in these docs are Vite-based, and Jazz works out of the box there with no `runtime` config.
- Expo / React Native uses the native Jazz runtime instead of the browser worker + Wasm asset path. Follow the Expo setup from the [Client quickstart](/docs/quickstarts/client#install) and you usually do not need `runtime`.
- If your browser bundle serves Jazz assets from a custom base path, rewrites module URLs to `file://...`, or otherwise fails to resolve `jazz_wasm_bg.wasm` / `jazz-worker.js`, pass `runtime` explicitly.

<Callout type="info">
  If you are not using the quickstart stack, start with the zero-config setup above. Add `runtime`
  only when your bundler or runtime cannot infer the Wasm and worker locations correctly.
</Callout>

## Client config options

<CreateJazzClientReference />

For React Native and Expo clients, make sure `serverUrl` points to a host your runtime can reach. For example, the Android emulator often needs `10.0.2.2` instead of `localhost`.

### Storage driver

By default, browser clients use `driver: { type: "persistent" }`, which runs a dedicated worker with OPFS-backed storage for offline support and local durability. Set `driver: { type: "memory" }` to skip the worker and OPFS entirely and keep data in memory only.

`memory` mode requires `serverUrl` to be set, since there is no local persistence.

## Runtime overrides

Use `runtime` when Jazz cannot infer where to load the browser worker or Wasm binary from, or when your runtime wants an in-memory Wasm source/module instead of URLs.

| Field                | Use it when                                                                 |
| -------------------- | --------------------------------------------------------------------------- |
| `runtime.baseUrl`    | Jazz runtime assets are served from a shared base path like `/assets/jazz/` |
| `runtime.wasmUrl`    | The Wasm file has an explicit public URL                                    |
| `runtime.workerUrl`  | The browser worker entry has an explicit public URL                         |
| `runtime.wasmSource` | Your runtime gives you Wasm bytes directly                                  |
| `runtime.wasmModule` | Your runtime gives you a precompiled `WebAssembly.Module`                   |

Jazz resolves these in this order:

1. `runtime.wasmModule`
2. `runtime.wasmSource`
3. `runtime.wasmUrl` / `runtime.workerUrl`
4. `runtime.baseUrl`
5. built-in zero-config fallback

### Browser asset overrides

React, Vue, and Svelte all pass the same config shape through to the shared client runtime. Plain TypeScript uses `createDb(...)` directly.

Use `baseUrl` when both assets live together under one public directory. Use explicit `wasmUrl` / `workerUrl` when your app serves them from different places.

### Edge-style Wasm setup

Some runtimes do not use the browser worker boot path at all. In those cases, prefer `runtime.wasmModule` or `runtime.wasmSource`.

```ts title="worker.ts"
import { createDb } from "jazz-tools";
import jazzWasmModule from "jazz-wasm/pkg/jazz_wasm_bg.wasm";

const db = await createDb({
  appId: "my-app",
  runtime: {
    wasmModule: jazzWasmModule,
  },
});
```

If your platform hands you raw bytes instead of a compiled module, use `runtime.wasmSource`:

```ts title="worker.ts"
const db = await createDb({
  appId: "my-app",
  runtime: {
    wasmSource: wasmBytes,
  },
});
```

See the [Cloudflare Wrangler example](https://github.com/garden-co/jazz2/tree/main/examples/cloudflare-worker-runtime-ts) for a complete `workerd` setup.
