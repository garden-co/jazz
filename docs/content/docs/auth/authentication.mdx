Jazz has three auth modes. `anonymous` and `demo` are for testing; most apps will need `external` authentication in production.

| Mode        | What it does                                            | When to use it                                   |
| ----------- | ------------------------------------------------------- | ------------------------------------------------ |
| `anonymous` | Generates one device token, persisted in `localStorage` | Prototyping, single-user apps, offline-only apps |
| `demo`      | Multiple synthetic user profiles per device             | Testing multi-user scenarios locally             |
| `external`  | Validates a JWT from your auth provider via JWKS        | Production apps with real user accounts          |

## Local auth for development

### Anonymous

Anonymous auth is the default when no JWT or backend auth is set up. Jazz generates a device token on first load and persists it in `localStorage`.

## External auth for production

Pass a JWT from your auth provider using the `jwtToken` option. Jazz validates it against your JWKS endpoint.

<Callout type="info" title="JWT? JWKS?">
  JWTs, or JSON Web Tokens are small JSON payloads signed by an authority&hairsp;—&hairsp;normally this is your authentication provider. Often this JSON payload includes details about your identity, but it can also include various other pieces of information known as 'claims'. Jazz can read the signed payload and use it in your permissions policies. To validate the signature and protect against spoofing, Jazz retrieves the certificate's public key from the JWKS endpoint you specify.

For more details, the [Auth0 docs](https://auth0.com/docs/secure/tokens/json-web-tokens) are a great starting point.

</Callout>

<Accordions type="single">
<Accordion title="Server configuration">

| Flag                | Purpose                                          | Development | Production                                                          |
| ------------------- | ------------------------------------------------ | ----------- | ------------------------------------------------------------------- |
| `--port`            | Port to listen on (defaults to 1625)             | Optional    | Optional                                                            |
| `--data-dir`        | Directory for persistent storage                 | Required    | Required                                                            |
| `--in-memory`       | Do not persist data (e.g. during development)    | Optional    | Optional                                                            |
| `--jwks-url <url>`  | JWKS endpoint for JWT signature verification     | Optional    | Optional, but required if using JWT auth to validate JWT signatures |
| `--backend-secret`  | Secret for backend session impersonation         | Optional    | Optional                                                            |
| `--admin-secret`    | Secret for admin operations (schema/policy sync) | Optional    | Optional                                                            |
| `--allow-anonymous` | Allow anonymous auth                             | Enabled     | Disabled                                                            |
| `--allow-demo`      | Allow demo auth                                  | Enabled     | Disabled                                                            |

Production mode activates when `NODE_ENV=production`.

<Accordion title="Session resolution">

When the server receives a request, it tries each auth method in priority order and uses the first match:

1. **Backend impersonation** &hairsp;—&hairsp; the request includes `X-Jazz-Backend-Secret` and `X-Jazz-Session` headers. A trusted backend service (e.g. a cron job or webhook handler) can act as any user by providing the shared secret and a base64-encoded session identifying the user to impersonate.
2. **External JWT** &hairsp;—&hairsp; the request includes an `Authorization: Bearer <jwt>` header. The token is validated against the JWKS endpoint configured with `--jwks-url`.
3. **Local auth** &hairsp;—&hairsp; the request includes `X-Jazz-Local-Mode` and `X-Jazz-Local-Token` headers. Used by anonymous and demo auth in development.
4. **No session** &hairsp;—&hairsp; none of the above matched.

Backend impersonation always takes precedence, so a backend service can reliably impersonate users even if the request also carries a JWT.

</Accordion>
</Accordions>
