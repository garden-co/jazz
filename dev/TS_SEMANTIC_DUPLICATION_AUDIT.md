# TS semantic duplication audit

Investigation slice: read-mostly audit of the TypeScript runtime/worker layers. No source changes made.

## Part 2: shortest path to native in-transaction reads

### What `ReadViewSpec` can express today

`ReadOpts` already carries `read_view: ReadViewSpec` as the semantic read view for one-shot reads (`crates/jazz/src/db.rs:5898`-`5911`). `ReadViewSpec` has a source, optional schema lens, and local overlays (`crates/jazz/src/protocol.rs:1253`-`1274`). Sources are `Current`, `Branch`, `MergedBranches`, and `Snapshot` (`crates/jazz/src/protocol.rs:1344`-`1376`). Overlays include `DirectBatch`, `AcceptedTransaction`, and `OpenTransaction` (`crates/jazz/src/protocol.rs:1402`-`1422`).

Execution support is narrower. `ensure_supported_read_view` accepts `Current`, and accepts a branch read view only when schema is default and overlays are empty (`crates/jazz/src/db.rs:6103`-`6113`). The query lowering path has the same shape: `query_read_set_for_read_view` supports current and branch-without-overlays, and rejects merged branches, snapshots, and branch overlays/lenses (`crates/jazz/src/node/query_eval.rs:2093`-`2142`). The wasm boundary rejects any provided `read_view`/`readView` before parsing options (`crates/jazz-wasm/src/lib.rs:2226`-`2268`).

There is already lower-level engine machinery for open transaction overlays. `tx_query_read_set` builds a read set with `SourceExpr::WithOverlays` and `OverlayRef::OpenTransaction(tx_id)` (`crates/jazz/src/node/query_eval.rs:1997`-`2048`). Source resolution accepts exactly an open transaction overlay on visible current/snapshot input (`crates/jazz/src/node/query_eval.rs:616`-`642`) and materializes it with `node.tx_current_rows(tx_id, table)` (`crates/jazz/src/node/query_eval.rs:850`-`864`). `NodeState::tx_query` evaluates a validated query inside an open exclusive transaction, records the predicate read, then applies core ordering/window finishing (`crates/jazz/src/node/query_eval.rs:7727`-`7760`).

### Where pending writes are authoritative today

The TS adapter is authoritative for public transaction IDs and read overlays. `beginTransaction` creates a `pendingTxs` entry with `writes: []` (`packages/jazz-tools/src/runtime/native-runtime/native-runtime-adapter.ts:657`-`660`). Transactional insert/update/upsert/delete paths stage writes both into the wasm transaction object and into `tx.writes` as `PendingTxWrite` rows/base/deletes (`packages/jazz-tools/src/runtime/native-runtime/native-runtime-adapter.ts:455`-`460`, `487`-`492`, `515`-`524`, `547`-`570`, `587`-`592`). `applyTransactionReadOverlay` then evaluates the read against committed rows plus that TS mirror (`packages/jazz-tools/src/runtime/native-runtime/native-runtime-adapter.ts:1135`-`1176`).

The wasm transaction object is not currently an open core transaction view. `WasmTx` stores `writes: Option<Vec<WasmTxWrite>>` in JS/Wasm memory (`crates/jazz-wasm/src/lib.rs:1049`-`1054`). `mergeableTx`, `mergeableTxForIdentity`, and `exclusiveTx` initialize that vector (`crates/jazz-wasm/src/lib.rs:1653`-`1679`). Transaction write methods append to it (`crates/jazz-wasm/src/lib.rs:1737`-`1823`), and `commit` consumes the vector and only then calls core commit helpers (`crates/jazz-wasm/src/lib.rs:1825`-`1847`). The returned `WasmWrite` carries a committed/committing `tx_id` and write-state handle (`crates/jazz-wasm/src/lib.rs:2184`-`2224`), but there is no wasm read entry that evaluates against the uncommitted vector.

So the shortest path is not just "pass `read_view` through wasm." Core has an open-exclusive transaction query path, but the current wasm API buffers pending writes outside that open-tx storage.

### Shortest viable implementation path

1. **Plumb an open transaction handle through wasm/napi.** Add a wasm transaction mode that opens a real core exclusive transaction immediately, keeps its `OpenTxId`/public tx id accessible, and applies each transactional write into core staging as it occurs. File-level targets: `crates/jazz-wasm/src/lib.rs` transaction constructors and `WasmTx` methods; matching napi/native runtime types if a node path exists outside wasm.

2. **Expose one-shot reads against that open transaction.** Add a wasm read method such as `allInTransaction(prepared, txHandle, opts)` or support a `read_view` with `OpenTransaction` only when it resolves to an open local tx. Internally call the existing core `tx_query` path for exclusive transactions rather than `Db::all` plus TS overlay. File-level targets: wasm read methods at `crates/jazz-wasm/src/lib.rs:1111`-`1124` and core `NodeState::tx_query` at `crates/jazz/src/node/query_eval.rs:7727`-`7760`.

3. **Make `ReadViewSpec` execution match the public model.** Longer-lived API shape should allow a read view like current source plus `overlays: [OpenTransaction { tx }]`. The query read-set resolver already has an overlay abstraction (`crates/jazz/src/node/query_eval.rs:616`-`642`), but public read-view execution currently rejects overlays (`crates/jazz/src/db.rs:6103`-`6113`; `crates/jazz/src/node/query_eval.rs:2115`-`2142`; wasm rejects at `crates/jazz-wasm/src/lib.rs:2258`-`2268`). This is mostly plumbing once open txs are stored in core rather than wasm-local vectors.

4. **Delete the TS overlay evaluator.** After the native read path exists, `NativeRuntimeAdapter.query()` can pass the transaction/read-view handle to core and remove `applyTransactionReadOverlay`, `rowMatchesQuery`, `sortRowsForQuery`, and the query-only TS comparison functions. The black-box contract should be "same query, same transaction, same rows/order as core."

Missing core/API pieces:

- Public wasm/napi transaction handles backed by core open transaction state, not `Vec<WasmTxWrite>` buffered in wasm memory.
- A supported public read option or method for current plus open-transaction overlay.
- Policy/read authorization evaluation for in-tx queries through the same core query program path as ordinary reads. `tx_query` currently uses `AuthorId::SYSTEM` in the call shown at `crates/jazz/src/node/query_eval.rs:7737`-`7743`; public session-aware transaction reads need the caller identity.
- Commit/rollback wiring from the new handle to existing exclusive/mergeable commit semantics, plus conflict/predicate-read preservation.
- TS adapter transaction bookkeeping simplified to lifecycle/transport only, with pending row mirrors deleted once read-your-writes is native.

Rough estimate:

- Plumbing-only prototype for exclusive transactions: 2-3 focused days if limited to one-shot `all`/`one` and local wasm memory/browser paths.
- Product-ready core/API path: 1-2 weeks to cover identity-aware reads, napi parity, error/lifecycle semantics, tests, and removal of TS overlay code.
- Relation subscriptions/read views over transaction overlays are a separate phase unless the immediate goal is only in-transaction one-shot reads.

## Draft SPEC open question

Proposed location: `crates/jazz/SPEC/13_db_api.md`. This is an API boundary principle more than a query-language feature; `6_queries.md` can cross-reference it.

> Open question: where may non-core layers evaluate query semantics?
>
> Principle: query semantics live in the core. TypeScript, wasm, napi, and runtime client layers may build queries, serialize query IR, cache prepared handles, transport frames, and hydrate typed application objects, but they must not independently evaluate predicates, ordering, limit/offset/windowing, relation/include membership, permission visibility, identity/dedupe, or semantic delta coalescing. When a client API needs read-your-writes or another alternate read view, the API should express that read view to core and let the normal lowered query execute there. Any TS-side reducer must be a specified wire-protocol reducer, not a second query engine.

## Tooling friction

A wasm/napi public API that exposes core `tx_query` for local tests would have made this audit shorter and would remove the current need to reason about two transaction staging layers.
